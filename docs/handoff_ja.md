## 30 秒サマリ

- ブラウザだけで、カメラを **時間窓で平均**してタイムラプス風の滑らかさを出し、**H.264 / MP4** にするデモ。

## 手元で動かす

`file://` ではモジュール import が失敗しやすい。**必ず HTTP(S)** で `index.html` を開く。

```bash
cd timeblend
python3 -m http.server 8080
```

ブラウザで `http://localhost:8080/` を開き、カメラを許可する。コマンド例の詳細は **`README.md`** の *Run locally* も参照。

**すぐ落ちるときの目安**: **WebGL2 非対応**または **`EXT_color_buffer_float` 無し**の環境では `setupWebGL` で失敗する（デスクトップの新しめ Chrome / Edge が無難）。

## 概要

ブラウザ上でカメラ映像を扱い、**フレーム間引きではなく時間方向の平均**で滑らかなタイムラプス風の動画を生成するデモ。

## コアコンセプト

| 従来のタイムラプス | Timeblend |
|-------------------|-----------|
| フレーム間引き → カクつきやすい | 一定窓内のフレームを加算し平均 → 滑らかさ・流れを強調 |

## 処理パイプライン

1. **getUserMedia**（`acquireCameraProbeStream`）：まず高めの `ideal` でプローブ。**既定は背面**（`environment`）。**背面が使えない／制約に合わない**ときは `NotFoundError` / `OverconstrainedError` の一定条件で **前面（`user`）に切り替えて再取得**（ノート等）。**音声は取らない**（`audio: false`）。→ トラックの **getCapabilities()** で幅・高さ上限を読み、**この端末では選べない出力プリセットを `disabled` にする**（`syncOutputPresetOptionsWithCapabilities`）
2. **選択中プリセットが無効** → 中止・メッセージ（**プルダウンの値を勝手に書き換えない**）。**`applyConstraints` が失敗** → 背面かつ取り直し可能なエラーなら **`start` 内で前面へ切替えて再試行**、それ以外または再試行後も失敗ならストリームを止め、**録画には進まず**メッセージのみ。
3. 問題なければ **WebGL2** でテクスチャ蓄積（`RGBA32F`、ピンポン）
4. **平均ウィンドウ**（1 秒〜60 分）ごとに平均をキャンバスへ描画
5. **WebCodecs** `VideoEncoder`（H.264）→ **mp4-muxer**（`esm.sh` の ESM import）で MP4 化

## UI / スタイル（方針）

- **OSS のスタイルを入れる場合の候補は、React 等のフレームワークを前提にしないものに絞る**（**CSS 中心・CDN で足せる・素の HTML と相性がよい**方向）。**採用は複数を比較したあとで決める。特定製品（例: Pico）一択ではない。**
- **対象外の例**: Ant Design、MUI、Chakra など **コンポーネント＝React（等）前提**のライブラリ。現状の **`index.html` + `app.js`（ビルドなし）**とは別レイヤーのコストになる。
- **利用規約など静的ページが増えても**、まずは **別 HTML（MPA）＋同じ CSS 土台**で足せる想定。
- **プレビュー枠**（`video` / `canvas` の黒背景・比率）は、どの土台でも **差分は `app.css` に残る**ことが多い。

## 主な機能（実装済み）

- 平均ウィンドウ: 1 / 3 / 5 / 10 / 30 / 60 **秒**、2 / 5 / 10 / 15 / 30 / 60 **分**
- 出力解像度: 横 1280×720 / 1920×1080 / 3840×2160、縦は長辺入れ替え
- MP4 ダウンロード・結果 `<video>` 再生
- 録画中は平均窓・解像度セレクト無効化
- **サンプリング**: `requestVideoFrameCallback` 優先、未対応は `setInterval`（100ms）
- **WebGL**: uniform のキャッシュ、`cleanup` から `teardownWebGL` でリソース解放（`gl.finish` は削除済み）
- **音声なし**: 録画・MP4 は **映像トラックのみ**（`getUserMedia` で `audio: false`）。

## 解像度・画質の前提

- カメラ実解像度がプリセットより小さい場合は **実質アップスケール** になりうる。ステータスにカメラ解像度を併記している。

## 性能（現状と論点）

- 負荷は主に **出力ピクセル数**（解像度）・**窓あたりの蓄積回数**・**エンコード**・**メモリ上の MP4** に依存する。
- **平均窓の emit** は **`setInterval`**。タブが **バックグラウンド**になると、ブラウザによって **`setInterval` / `setTimeout` が間引かれたり最小間隔が伸びたり**することがあり、**長い窓ほど「設定秒数どおりに発火しない」**ことがある（実装は `app.js` の `emitTimer`）。
- 蓄積は **全画面フラグメントシェーダ**で、ピクセル間に依存がない（加算は各ピクセル独立）。GPU はもともとピクセル並列なので、**画面をタイル分割して複数回描画するだけでは、処理フラグメント総数が変わらない限り劇的な高速化は期待しにくい**（帯域・中間バッファの制約下では別議論になりうるが、要計測）。
- **効果が読みやすい改善**の例: 解像度・ビットレートの見直し、長窓のタイマー設計、ストリーミング mux、WebGPU への移行検討など。

### 出力 MP4 とメモリ（現状と「ディスク直書き」）

- **現状**: `mp4-muxer` の **`ArrayBufferTarget`** で、finalize まで **完成 MP4 を JS ヒープ上に溜める**。`Muxer` には **`fastStart: "in-memory"`** が指定されており、完成形をメモリ上で組み立てる前提と整合する。出力が長いほど **RAM 占有が増え**、長尺では落ちやすい。
- **メモリに溜めずストレージへ直接、は可能か**: **「ブラウザだから絶対無理」ではない**。ただし **任意パスへの無許可書き込みはできない**。現実的には **File System Access API**（例: `showSaveFilePicker` → **`WritableStream` に逐次書き込み**）で、**ユーザーが保存先ファイルを選んだうえで**チャンクを流す形になる。**Chromium 系が主戦場で、Safari 等は要確認**。
- **mux 側**: 従来型 MP4 で **末尾に `moov` をまとめる方式**だと、**ストリーミング書き出しと相性が悪い**ことがある。**フラグメント MP4（fMP4）**のように **チャンク単位で確定できる形式**に寄せると、ディスク追記と組み合わせやすい（**ライブラリ・ターゲットの対応を要調査**）。**いまの `ArrayBufferTarget` 前提のままでは RAM 回避にはならない**。
- **当面の緩和（実装しやすい順）**: **出力の最大時間または最大フレーム数**で打ち切る、ビットレート・解像度を抑える、サイト上で上限と注意を書く、など。

## 公開・運用（README と併せて）

- **運用方針（担当者合意）**: **リポジトリはプライベート**を前提とする。**ホスティングに一定のコスト**がかかることは許容する。無料の **GitHub Pages だけ**に寄せると **プライベート可否がプラン・仕様に依存**するため、必要なら **Cloudflare Pages / S3 + CloudFront 等**へ寄せてソースと配信を分ける。**公開 URL に載る HTML/JS は誰でも取得できる**（リポジトリを隠してもクライアントは丸見え）点は変わらない。
- 本番 URL は **HTTPS**（カメラのため）。**静的ファイルを配るだけ**でよい（サーバ処理なし）。
- **公開先の例**: 会話時点では **GitHub Pages** を利用している想定。実際の URL は **`README.md` に書く**か、末尾 **[作業中断メモ](#作業中断メモ)** に残すと再開しやすい。**Cloudflare Pages** や **S3 + CloudFront** も同様に妥当（詳細手順は本書では扱わない）。
- **`index.html` の Cache-Control メタ**（`no-store` 等）により、ブラウザは更新を取りにくくしやすい。CDN と併用する場合は **キャッシュ挙動**を意識する。
- **Content-Security-Policy** を厳しくする場合、ESM 取得のため **`https://esm.sh`** を `script-src` / `connect-src` 等に含められるか要確認（未設定ならブラウザ既定のまま）。
- 依存 CDN（`esm.sh`）・ブラウザ内処理のみであることの説明は **`README.md`** の Privacy / Limitations を参照。公開前チェックも README と **[直近で必要になりやすいこと](#直近で必要になりやすいこと)** を併用。
