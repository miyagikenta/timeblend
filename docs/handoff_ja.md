# Timeblend — 引き継ぎ・仕様メモ（日本語）

この文書は、プロジェクトの意図・現状実装・運用上の注意を引き継ぐための資料です。  
実装のソースオブトゥルースはリポジトリ内の **`index.html`**（単一ファイルに HTML / CSS / JS が同居）です。

---

## 概要

ブラウザ上でカメラ映像を扱い、**フレーム間引きではなく時間方向の平均**で滑らかなタイムラプス風の動画を生成するデモです。

## コアコンセプト

| 従来のタイムラプス | Timeblend |
|-------------------|-----------|
| フレーム間引き → カクつきやすい | 一定窓内のフレームを加算し平均 → 滑らかさ・流れを強調 |

## 処理パイプライン

1. **getUserMedia** でカメラ取得（解像度プリセットに合わせた `ideal` 制約）
2. **WebGL2** でビデオをテクスチャ化し、`RGBA32F` の蓄積バッファへシェーダで加算
3. **ピンポン**（蓄積用テクスチャ 2 枚）で上書き競合を防止
4. **平均ウィンドウ**（1〜60 秒、UI で選択）ごとに平均シェーダでキャンバスへ描画
5. **WebCodecs** `VideoEncoder`（H.264）でフレーム化
6. **mp4-muxer**（`esm.sh` 経由の ESM import）で MP4 化し、再生・ダウンロード

## 技術スタック（現状）

| 用途 | 技術 |
|------|------|
| UI / 配線 | 素の HTML / CSS / JS（`type="module"`） |
| カメラ | MediaDevices / `getUserMedia` |
| 蓄積・表示 | WebGL2、`EXT_color_buffer_float` 必須 |
| エンコード | WebCodecs（H.264、`VideoEncoder.isConfigSupported` で利用可能な `avc1.*` を順に試行） |
| コンテナ | mp4-muxer 5.x（CDN） |
| バージョン表示 | `index.html` 内の **`VERSION`** 定数（手動更新） |

## 主な機能（実装済み）

- リアルタイムの蓄積とプレビュー（キャンバス）
- **平均ウィンドウ** 1 / 3 / 5 / 10 / 30 / 60 秒
- **出力解像度プリセット**  
  - 横持ち: 1280×720 / 1920×1080 / 3840×2160  
  - 縦持ち: 720×1280 / 1080×1920 / 2160×3840（長辺に合わせて入れ替え）
- **MP4** ダウンロード・結果 `<video>` 再生
- 録画中は平均窓・解像度のセレクトを無効化

## 実装メモ（保守で効くところ）

### サンプリング

- **`requestVideoFrameCallback`** を優先し、カメラの表示フレームに同期して蓄積
- 未対応環境では **`setInterval`（100ms 間隔）** にフォールバック

### WebGL

- **uniform location** はプログラムリンク後に一度だけ取得しキャッシュ（ホットパスで毎回 `getUniformLocation` しない）
- **`cleanup()`** から **`teardownWebGL()`** を呼び、プログラム・テクスチャ・FBO・VAO を `delete*` してから `gl = null`。連続録画時の GPU リソースリークを防ぐ意図
- 以前入っていた **`gl.finish()`** は削除済み（`VideoFrame(canvas)` 前の過剰同期の削減）。不具合が出たら再検討可

### 解像度・画質の前提

- 設計としては「処理解像度＝出力解像度」を揃えているが、**カメラが返す実解像度がプリセットより小さい場合**は、フルスクリーンクワッド描画により **実質アップスケール** になりうる。UI ステータスにカメラ実解像度を併記している
- ビットレートは出力ピクセル数に応じた段階（720p / 1080p / 4K 目安）

### 単一ファイルの利点・注意

- **ビルド不要**で配布しやすい一方、**ES モジュール**のため、多くのブラウザでは **`file://` 直開きより HTTP で配信**した方が安全（mp4-muxer の import のため）
- 規模が増えたら **`main.js` 分離 + 静的サーバー** を検討（別途判断）

## パフォーマンス前提

- ピクセル数に比例して GPU・エンコード負荷が増える
- **4K** は特に重い（端末・ブラウザ依存大）
- 長時間録画: 発熱・メモリ・エンコードキューがボトルネックになりやすい
